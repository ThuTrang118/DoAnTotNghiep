VAI TRÒ
- Đây là tài liệu kiến thức chung để AI suy ra coverage kiểm thử từ đặc tả nghiệp vụ.
- Tài liệu này chỉ hướng dẫn:
  + cách đọc đặc tả
  + cách nhận diện constraint
  + cách chọn kỹ thuật kiểm thử
  + cách tách coverage item
  + cách sinh testcase đúng bản chất
- Tài liệu này KHÔNG sinh JSON output.
- Tài liệu này KHÔNG thay thế mô tả nghiệp vụ riêng của từng chức năng.
- Tài liệu này không được thiên về bất kỳ chức năng cụ thể nào.

==================================================================================
I. NGUYÊN TẮC NỀN TẢNG
==================================================================================

1. Chỉ được suy luận từ đặc tả đã cho:
- field input
- constraint
- expected
- quan hệ giữa các field
- điều kiện thành công / thất bại

2. Không được:
- tự thêm constraint
- tự thêm nghiệp vụ không có trong mô tả
- tự bịa expected
- tự giả định format, độ dài, miền giá trị nếu đặc tả không nói

3. Mỗi testcase phải:
- có đúng 1 mục tiêu kiểm thử chính
- có input cụ thể
- có expected cụ thể
- truy ngược được về 1 rule hoặc 1 hành vi rõ ràng trong đặc tả

4. Với testcase âm tính:
- chỉ được có đúng 1 lỗi chính
- các field không phải mục tiêu lỗi phải dùng dữ liệu hợp lệ

5. Không được gộp:
- nhiều lỗi vào 1 testcase
- nhiều invalid class khác bản chất vào 1 testcase
- nhiều boundary point vào 1 testcase

==================================================================================
II. PHÂN LOẠI COVERAGE THEO FIELD
==================================================================================

Với MỖI field, AI phải tự phân loại coverage theo 4 nhóm trước khi sinh testcase:

1. REQUIRED
- field có bắt buộc hay không
- nếu có expected khi rỗng -> tạo 1 testcase EMPTY riêng

2. INVALID CLASS
- liệt kê từng lớp lỗi độc lập có expected xác định được
- mỗi lớp lỗi độc lập = 1 coverage item riêng

3. BOUNDARY
- chỉ xét nếu field có biên định lượng rõ ràng trong đặc tả
- mỗi boundary point = 1 coverage item riêng

4. RELATION
- nếu field có quan hệ với field khác và quan hệ đó có expected rõ ràng
- tạo 1 coverage item riêng cho lỗi quan hệ

Chỉ sau khi phân loại xong từng field mới được hợp nhất thành bộ testcase cuối cùng.

==================================================================================
III. VALID / EMPTY / INVALID
==================================================================================

Mỗi field cần được phân tích theo 3 trạng thái:

1. VALID
- giá trị thỏa tất cả constraint liên quan

2. EMPTY / MISSING
- không nhập dữ liệu
- để trống
- chưa chọn giá trị

3. INVALID
- có dữ liệu nhưng sai

INVALID có thể gồm:
- sai độ dài
- sai format
- sai kiểu dữ liệu
- sai prefix / suffix
- sai pattern
- sai miền giá trị
- sai tập ký tự cho phép
- sai quan hệ với field khác
- sai logic nghiệp vụ

QUY TẮC:
- EMPTY khác INVALID
- không gộp EMPTY vào INVALID
- mỗi invalid class chỉ được thành testcase riêng nếu class đó có expected xác định được từ đặc tả hoặc thuộc một expected chung đủ rõ

==================================================================================
IV. DECISION TABLE (DT)
==================================================================================

DT dùng khi bản chất kiểm thử là:
- field bắt buộc
- điều kiện - kết quả
- quan hệ giữa nhiều field
- workflow logic
- tổ hợp trạng thái dẫn tới kết quả khác nhau

KHÔNG dùng DT cho:
- boundary
- format class
- một lớp dữ liệu đơn

==================================================================================
V. EQUIVALENCE PARTITIONING (EP)
==================================================================================

EP dùng khi:
- dữ liệu được chia thành các lớp tương đương

QUY TẮC:
- mỗi invalid class = 1 testcase
- không gộp class

==================================================================================
V.1. SAME EXPECTED ≠ SAME TESTCASE
==================================================================================

Nếu nhiều invalid class khác nhau cùng dẫn tới một expected:

→ PHẢI tách riêng từng testcase

==================================================================================
VI. BOUNDARY VALUE ANALYSIS (BVA)
==================================================================================

BVA CHỈ được dùng khi đặc tả có ràng buộc định lượng rõ ràng:

- từ min đến max
- đúng N
- tối đa N
- tối thiểu N

NẾU KHÔNG CÓ:
→ KHÔNG được dùng BVA

1. ĐỊNH NGHĨA MIN / MAX / N

- min: giá trị nhỏ nhất hợp lệ
- max: giá trị lớn nhất hợp lệ
- N: giá trị cố định phải đạt

QUAN TRỌNG:
- Đây là MỐC GỐC
- reference_value phải là mốc này
- KHÔNG được dùng giá trị đã lệch biên làm reference_value

2. ĐỊNH NGHĨA ĐIỂM BIÊN

- min-1: nhỏ hơn min đúng 1 đơn vị
- min: bằng min
- max: bằng max
- max+1: lớn hơn max đúng 1 đơn vị

- N-1: nhỏ hơn N đúng 1 đơn vị
- N: bằng N
- N+1: lớn hơn N đúng 1 đơn vị

"1 đơn vị" phụ thuộc kiểu dữ liệu:
- chuỗi → 1 ký tự
- số → 1 đơn vị số

3. XÁC ĐỊNH BIÊN

Range: min-1, min, max, max+1

Fixed: N-1, N, N+1

4. BIÊN TƯỜNG MINH ⇒ BVA BẮT BUỘC

Nếu có biên:
→ PHẢI dùng BVA

KHÔNG được:
- thay bằng EP
- gộp thành 1 testcase

5. BVA SCHEMA LOGIC

Fixed:
- rule_type = fixed_length
- reference_value = N
- boundary = N-1, N, N+1

Range:
- rule_type = range
- reference_value = min hoặc max
- boundary = min-1, min, max, max+1

CẤM:
- sai rule_type
- reference_value không phải mốc gốc

6. SINH DỮ LIỆU BIÊN

- N-1 → input phải đúng N-1
- N → input phải đúng N
- N+1 → input phải đúng N+1

CẤM:
- boundary ≠ input

7. VALID & INVALID BOUNDARY

- min, max → VALID
- min-1, max+1 → INVALID

QUAN TRỌNG:
- BVA phải có cả valid và invalid boundary khi cần

8. CÔ LẬP BVA
- chỉ sai biên
- rule khác phải đúng

9. BẮT BUỘC ĐỦ COVERAGE BVA
Nếu tồn tại biên:

→ PHẢI liệt kê ĐẦY ĐỦ tất cả boundary point theo đúng loại:

Range:
- min-1
- min
- max
- max+1

Fixed:
- N-1
- N
- N+1

QUY TẮC BẮT BUỘC:
- Mỗi boundary point = 1 coverage item riêng
- Không được thiếu bất kỳ điểm nào
- Không được gộp nhiều điểm vào 1 testcase

10. ÁNH XẠ BOUNDARY → INPUT
Input phải KHỚP CHÍNH XÁC với boundary_point:

- N-1 → độ dài = N-1
- N → độ dài = N
- N+1 → độ dài = N+1

CẤM:
- ghi boundary_point nhưng input không đúng
