PHÂN TÍCH CHỨC NĂNG ĐĂNG NHẬP

Bạn hãy phân tích chức năng LOGIN của website MWC và sinh ra các nhóm hành vi kiểm thử theo phương pháp kiểm thử hộp đen.

==================================================
1. MÔ TẢ CHỨC NĂNG
==================================================

Form đăng nhập có 2 trường:

- Username
- Password

Người dùng nhập dữ liệu và gửi form để xác thực tài khoản.

==================================================
2. RÀNG BUỘC DỮ LIỆU
==================================================

Username:
- bắt buộc
- tối đa 25 ký tự

Password:
- bắt buộc
- độ dài từ 8 đến 20 ký tự
- phải chứa tối thiểu:
  - 1 chữ hoa
  - 1 chữ thường
  - 1 chữ số
  - 1 ký tự đặc biệt

==================================================
3. TÀI KHOẢN HỢP LỆ
==================================================

Tài khoản hợp lệ duy nhất dùng để kiểm thử:

Username: AnhDuong11
Password: anhduong@123

==================================================
4. QUY TẮC HỆ THỐNG
==================================================

Nếu Username hoặc Password là chuỗi rỗng ""
=> Expected: Vui lòng điền vào trường này

Nếu Username và Password đều có dữ liệu nhưng không khớp chính xác tài khoản hợp lệ
=> Expected: Tên đăng nhập hoặc mật khẩu không đúng!

Nếu Username = "AnhDuong11" và Password = "anhduong@123"
=> Expected: AnhDuong11

==================================================
5. QUY TẮC SO SÁNH
==================================================

Username và Password được so sánh chính xác tuyệt đối.

Chỉ cần khác một ký tự cũng được xem là sai.

Các trường hợp sau đều được xem là dữ liệu sai:

- chỉ chứa khoảng trắng
- có khoảng trắng đầu hoặc cuối
- khác chữ hoa chữ thường

Chuỗi chỉ chứa khoảng trắng KHÔNG phải chuỗi rỗng.

==================================================
6. MỤC TIÊU SINH DỮ LIỆU
==================================================

Hãy áp dụng các kỹ thuật kiểm thử hộp đen sau để suy luận các nhóm hành vi kiểm thử:

- Equivalence Partitioning
- Boundary Value Analysis
- Error Guessing
- Cause–Effect reasoning

Từ các ràng buộc dữ liệu và rule hệ thống, hãy sinh ra các trường hợp kiểm thử đại diện cho:

- dữ liệu hợp lệ
- dữ liệu thiếu
- dữ liệu vượt giới hạn
- dữ liệu ở giá trị biên
- dữ liệu vi phạm quy tắc định dạng
- các lỗi nhập liệu thường gặp

Chỉ sinh các trường hợp cần thiết để bao phủ các hành vi khác biệt thật sự.

Không cần sinh các biến thể lặp lại nếu chúng không làm tăng coverage.

==================================================
7. QUY TẮC CHỐNG TRÙNG
==================================================

Không được tạo 2 item có cùng:

- UsernamePattern
- PasswordPattern
- Expected

Mỗi item phải đại diện cho một hành vi kiểm thử khác nhau.

==================================================
8. QUY TẮC EXPECTED
==================================================

Chỉ được dùng đúng 3 giá trị Expected sau:

1. Vui lòng điền vào trường này
2. Tên đăng nhập hoặc mật khẩu không đúng!
3. AnhDuong11

Không được để Expected rỗng.

==================================================
9. ĐỊNH DẠNG OUTPUT
==================================================

Chỉ trả về DUY NHẤT một JSON object hợp lệ theo mẫu:

{
  "items": [
    {
      "GroupID": "LGG01",
      "BehaviorGroup": "Đăng nhập hợp lệ",
      "UsernamePattern": "AnhDuong11",
      "PasswordPattern": "anhduong@123",
      "Expected": "AnhDuong11"
    }
  ]
}

==================================================
10. QUY TẮC OUTPUT
==================================================

- Chỉ JSON
- Không markdown
- Không code block
- Không giải thích
- Không viết chữ trước JSON
- Không viết chữ sau JSON

Mỗi item chỉ có đúng 5 trường:

GroupID
BehaviorGroup
UsernamePattern
PasswordPattern
Expected

==================================================
11. KIỂM TRA TRƯỚC KHI TRẢ KẾT QUẢ
==================================================

Trước khi trả kết quả, tự kiểm tra:

- mọi item đều đủ 5 trường
- JSON đóng đủ ngoặc
- không có item trùng
- các giá trị Expected hợp lệ
